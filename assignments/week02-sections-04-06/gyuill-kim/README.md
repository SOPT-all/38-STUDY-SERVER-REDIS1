# Week 02 - 김규일

## Redis 캐싱 전략과 성능 이해

이번 주차에서는 Redis를 단순히 "빠른 저장소"로 보는 것을 넘어서 왜 Redis가 빠른지와 어떤 상황에서 캐싱을 적용해야 하는지를 중심으로 정리해봤습니다.

2주차 범위는 크게 세 가지로 볼 수 있습니다.

1. Redis 캐싱 전략
2. 애플리케이션에서 Redis를 캐시로 사용하는 방식
3. 캐싱을 적용할 때 생기는 문제와 운영상 주의점

---

## 발표 핵심 메시지

Redis 캐싱의 핵심은 "느린 원본 저장소에 매번 접근하지 않고 더 빠른 계층에 데이터를 임시로 저장해 응답 속도를 높이는 것"입니다.

하지만 캐싱은 공짜 성능 개선이 아닙니다.  
응답 속도를 얻는 대신 데이터 일관성, 메모리 비용, 만료 정책, 장애 대응 같은 새로운 고민이 생깁니다.

따라서 Redis를 잘 쓴다는 것은 단순히 `GET`, `SET`을 잘 쓰는 것이 아니라 다음 질문에 답할 수 있다는 뜻입니다.

- 어떤 데이터를 캐싱할 것인가?
- 언제 캐시를 읽고 언제 DB를 읽을 것인가?
- 원본 데이터가 바뀌면 캐시는 어떻게 갱신할 것인가?
- 캐시가 사라지거나 Redis가 장애 나면 서비스는 어떻게 동작할 것인가?

---

## 1. 캐시와 캐싱

### 캐시(Cache)

캐시는 원본 데이터보다 빠르게 접근할 수 있는 임시 저장소입니다.

웹 애플리케이션에서 원본 데이터는 보통 RDB나 외부 API입니다.  
이 원본 데이터에 매번 접근하면 네트워크, 디스크 I/O, SQL 처리, 락 경합 같은 비용이 발생합니다.

캐시는 이 비용을 줄이기 위해 자주 쓰이는 데이터를 더 가까운 곳에 저장합니다.

예를 들면 다음과 같습니다.

- CPU는 메인 메모리보다 빠른 CPU Cache를 사용함
- 브라우저는 이미지, CSS, JS 파일을 로컬에 캐싱함
- 서버는 자주 조회되는 DB 결과를 Redis에 캐싱함

즉 캐시는 특정 기술 이름이 아니라 "느린 계층 앞에 빠른 계층을 두는 설계 방식"입니다.

### 캐싱(Caching)

캐싱은 데이터를 캐시에 저장하고, 이후 요청에서 원본 대신 캐시를 먼저 사용하는 방식입니다.

실무에서 "이 API 캐싱하자"는 말은 보통 다음 의미입니다.

> 같은 요청이 반복될 때 매번 DB나 외부 API를 호출하지 말고, Redis 같은 빠른 저장소에서 바로 응답하자.

캐싱의 목적은 크게 세 가지입니다.

- 응답 시간 단축
- DB 또는 외부 API 부하 감소
- 트래픽 증가 상황에서 시스템 안정성 확보

## 2. 왜 Redis는 빠른가?

### 2-1. 메모리 기반 저장소

Redis가 빠른 가장 큰 이유는 데이터를 주로 RAM에 저장하기 때문입니다.  
하지만 Redis가 빠른 이유를 단순히 "RAM을 사용하기 때문"이라고만 설명하면 부족합니다.

Redis 캐싱의 성능은 여러 계층의 비용을 줄이는 데서 나옵니다.

- 디스크 I/O를 줄인다.
- SQL 처리 비용을 줄인다.
- 복잡한 인덱스 탐색을 단순 Key 조회로 바꾼다.
- DB 커넥션과 트랜잭션 부담을 줄인다.
- 자주 반복되는 계산과 직렬화 비용을 줄인다.
- 원본 저장소 앞에서 트래픽을 흡수한다.

즉 Redis는 "메모리를 쓰는 빠른 저장소"이기도 하지만, 더 정확히는 "느린 경로를 빠른 경로로 우회하게 해주는 캐시 계층"입니다.

컴퓨터는 저장 공간을 하나만 쓰지 않습니다.  
속도와 용량, 비용이 다른 여러 저장 계층을 함께 사용합니다.

대략적인 저장 계층별 접근 속도는 아래처럼 이해할 수 있습니다.

| 계층 | 대략적인 접근 속도 | 용량 | 특징 |
| --- | --- | --- | --- |
| Register | 1ns 이하 | 매우 작음 | CPU 내부에서 직접 사용 |
| L1 Cache | 약 1ns | 매우 작음 | CPU 코어와 가장 가까운 캐시 |
| L2/L3 Cache | 수 ns ~ 수십 ns | 작음 | CPU와 RAM 사이의 캐시 |
| RAM | 약 100ns | GB 단위 | 일반적인 메인 메모리 |
| SSD | 약 100us | GB ~ TB 단위 | 디스크보다 빠르지만 RAM보다 느림 |
| HDD | 약 10ms | TB 단위 | 물리적인 회전과 헤드 이동 비용 존재 |

단위 차이를 보면 RAM과 SSD 사이에도 큰 차이가 있습니다.

- `1us` = `1,000ns`
- `1ms` = `1,000us` = `1,000,000ns`

즉 RAM은 디스크보다 훨씬 빠른 계층입니다.  
Redis는 이 빠른 계층을 적극적으로 활용하기 때문에 짧은 응답 시간을 만들 수 있습니다.

발표에서는 저장 계층을 이렇게 비유할 수도 있습니다.

```text
CPU Cache: 책상 위 메모지
RAM: 책상 서랍
SSD/HDD: 창고
Network: 다른 건물에 있는 창고
```

가까운 곳에 있는 데이터일수록 빠르게 가져올 수 있습니다.

RAM은 Random Access Memory입니다.  
이름 그대로 임의의 주소에 비교적 균일한 시간으로 접근할 수 있습니다.

디스크는 데이터를 읽을 때 저장 장치 컨트롤러, 파일 시스템, 블록 I/O, OS 스케줄링 같은 여러 단계를 거칩니다.  
반면 RAM 접근은 CPU가 메모리 주소를 통해 훨씬 짧은 경로로 데이터를 읽고 씁니다.

물론 RAM 접근도 완전히 공짜는 아닙니다.  
CPU 입장에서는 RAM도 느리기 때문에 CPU는 L1, L2, L3 Cache를 사용해 RAM 접근 횟수를 줄입니다.

하지만 애플리케이션 관점에서는 디스크나 네트워크에 비해 RAM은 매우 빠른 계층입니다.  
Redis는 데이터를 메모리에 올려두고 Key 기반으로 접근하므로 반복 조회에서 큰 이점을 얻습니다.

### 2-2. RDB 조회가 Redis보다 무거운 이유

RDB가 느리다는 말은 RDB가 나쁜 기술이라는 뜻이 아닙니다.  
RDB는 정확성, 트랜잭션, 복잡한 조회, 관계 표현을 위해 많은 일을 합니다.

RDB는 Redis보다 많은 기능을 제공합니다.

- SQL
- Join
- Transaction
- Lock
- MVCC
- Index
- Constraint
- Query Optimizer
- Persistence
- Recovery

이 기능들은 데이터 정합성과 복잡한 질의를 위해 필요합니다.  
하지만 기능이 많다는 것은 조회 경로도 길어진다는 뜻입니다.

일반적인 RDB 조회 과정을 단순화하면 다음과 같습니다.

```text
Application
  -> DB Connection 획득
  -> SQL 전송
  -> SQL Parser
  -> Query Optimizer
  -> Execution Plan 선택
  -> Index 탐색
  -> Buffer Pool 확인
  -> 필요 시 Disk I/O
  -> Transaction / Lock / MVCC 처리
  -> 결과 Row 생성
  -> Network 전송
  -> Application 객체로 변환
```

반면 Redis는 보통 단순한 Key-Value 접근을 합니다.

```text
Application
  -> Redis에 key 조회
  -> value 반환
```

물론 Redis도 네트워크 통신, 명령 파싱, 자료구조 접근, 직렬화/역직렬화 비용이 있습니다.  
그래도 RDB의 복잡한 조회 경로에 비하면 훨씬 단순합니다.

Redis 캐싱은 이 차이를 이용합니다.

```text
복잡한 SQL 조회 결과
  -> 한 번 계산
  -> Redis에 key-value 형태로 저장
  -> 다음 요청부터 단순 key 조회
```

즉 Redis 캐싱은 DB 자체를 빠르게 만드는 기술이라기보다, DB까지 가지 않아도 되는 요청을 늘려 전체 시스템의 평균 응답 시간을 낮추는 기술입니다.

### 2-3. DB도 메모리를 사용한다

중요한 점은 DB도 내부적으로 메모리를 사용한다는 것입니다.

예를 들어 MySQL의 InnoDB Buffer Pool은 자주 접근하는 데이터 페이지와 인덱스 페이지를 메모리에 올려둡니다.  
그래서 DB 조회가 항상 디스크를 읽는 것은 아닙니다.

```text
DB 조회
  -> Buffer Pool에 있음: 메모리에서 읽음
  -> Buffer Pool에 없음: 디스크에서 읽고 Buffer Pool에 올림
```

그럼에도 Redis 캐시가 의미 있는 이유는 다음과 같습니다.

1. DB의 메모리 캐시는 페이지 단위이고, Redis 캐시는 애플리케이션 결과 단위일 수 있다.
2. DB는 SQL 실행, 인덱스 탐색, 트랜잭션 처리 비용이 여전히 존재한다.
3. Redis는 반복되는 조회 결과를 애플리케이션이 원하는 형태로 바로 저장할 수 있다.
4. Redis는 DB 앞단에서 요청 자체를 줄여준다.

즉 Redis가 빠른 이유를 "RAM이라서 빠르다"로만 끝내면 부족합니다.  
Redis 캐싱의 진짜 효과는 복잡한 원본 조회 과정을 단순한 캐시 조회로 바꾸는 데 있습니다.

### 2-4. Page Cache와 Buffer Pool

운영체제도 디스크 I/O를 줄이기 위해 Page Cache를 사용합니다.

Page Cache는 디스크에서 읽은 파일 데이터를 RAM에 보관합니다.  
같은 파일 블록을 다시 읽을 때 디스크가 아니라 메모리에서 읽을 수 있습니다.

DB에는 자체 Buffer Pool이 있습니다.  
DB는 테이블과 인덱스를 페이지 단위로 관리하고, 자주 쓰는 페이지를 Buffer Pool에 올립니다.

정리하면 다음과 같습니다.

| 계층 | 캐시 대상 | 관리 주체 |
| --- | --- | --- |
| CPU Cache | 메모리의 작은 블록 | CPU |
| OS Page Cache | 파일/디스크 블록 | 운영체제 |
| DB Buffer Pool | DB 페이지와 인덱스 페이지 | DB 엔진 |
| Redis Cache | 애플리케이션이 정한 key-value | 애플리케이션 + Redis |

Redis 캐싱은 운영체제나 DB가 자동으로 해주는 캐시와 다릅니다.  
애플리케이션이 "이 결과는 재사용할 가치가 있다"고 판단한 데이터를 명시적으로 저장합니다.

자동 캐시는 하위 계층의 데이터를 빠르게 읽게 해줍니다.  
Redis 캐시는 애플리케이션 관점의 결과를 빠르게 재사용하게 해줍니다.

### 2-5. 디스크 I/O 비용

디스크 I/O는 크게 Sequential I/O와 Random I/O로 나눌 수 있습니다.

Sequential I/O는 연속된 데이터를 순서대로 읽고 쓰는 방식입니다.  
디스크나 SSD가 비교적 효율적으로 처리할 수 있습니다.

예를 들면 로그 파일을 순서대로 쓰는 작업입니다.

Random I/O는 서로 떨어진 위치의 데이터를 여기저기 읽는 방식입니다.  
인덱스를 타고 여러 데이터 페이지를 조회할 때 Random I/O가 발생할 수 있습니다.

HDD에서는 Random I/O가 특히 비쌉니다.  
헤드 이동과 회전 대기 시간이 발생하기 때문입니다.

SSD는 HDD보다 Random I/O에 강하지만, 그래도 RAM 접근보다는 훨씬 느립니다.

RDB는 인덱스를 사용하더라도 상황에 따라 여러 페이지를 읽어야 합니다.  
이때 Buffer Pool에 없는 페이지는 디스크에서 가져와야 하므로 지연이 커질 수 있습니다.

Redis 캐시는 이런 디스크 접근 가능성을 줄입니다.

### 2-6. 네트워크 비용도 포함해서 봐야 한다

Redis가 빠르다고 해도 애플리케이션과 Redis가 다른 프로세스 또는 다른 서버에 있다면 네트워크 비용이 생깁니다.

```text
Application
  -> Network
  -> Redis
  -> Network
  -> Application
```

같은 서버 내부 통신보다 다른 서버로 가는 네트워크 통신이 더 비쌉니다.  
클라우드 환경에서는 같은 VPC 안에 있어도 네트워크 지연이 존재합니다.

그래서 Redis 조회가 무조건 CPU Cache나 RAM 접근 속도만큼 빠른 것은 아닙니다.  
애플리케이션 입장에서는 다음 비용을 모두 포함해서 봐야 합니다.

- 네트워크 왕복 시간
- Redis 명령 처리 시간
- 직렬화/역직렬화 시간
- 커넥션 풀 대기 시간
- 애플리케이션 객체 변환 시간

그럼에도 Redis 캐시가 효과적인 이유는 DB 조회 전체 비용보다 Redis 조회 전체 비용이 낮은 경우가 많기 때문입니다.

정리하면 다음과 같습니다.

> Redis 자체는 메모리 기반이라 빠르지만, 애플리케이션에서 체감하는 속도는 네트워크와 직렬화 비용까지 포함한 값이다.

### 2-7. Redis의 단일 스레드 이벤트 루프

Redis는 명령 실행을 주로 단일 스레드 이벤트 루프 모델로 처리하는 것으로 알려져 있습니다.

단일 스레드라는 말만 들으면 느릴 것 같지만, Redis의 작업 특성과 잘 맞습니다.

Redis 명령은 대부분 다음 특징을 가집니다.

- 메모리 접근 중심
- 짧은 작업 시간
- 단순한 자료구조 연산
- 디스크 대기 시간이 적음
- 락 경합이 적음

멀티스레드 프로그램에서는 여러 스레드가 같은 데이터에 접근할 때 락이 필요합니다.  
락은 동시성 문제를 해결하지만, 락 경합과 Context Switching 비용을 만들 수 있습니다.

Redis는 단일 스레드 이벤트 루프를 사용해 명령을 순차적으로 처리함으로써 복잡한 락을 줄입니다.

```text
Client 요청들
  -> Event Loop
  -> 명령을 빠르게 하나씩 처리
  -> 응답
```

이 구조는 짧고 빠른 명령이 많을 때 강합니다.

하지만 하나의 명령이 오래 걸리면 다른 요청도 기다립니다.  
예를 들어 큰 키를 한 번에 처리하거나, 많은 데이터를 한 번에 조회하거나, 무거운 Lua Script를 실행하면 Redis 전체 응답에 영향을 줄 수 있습니다.

그래서 Redis에서는 다음을 조심해야 합니다.

- 너무 큰 key/value
- 너무 큰 collection
- 운영 환경에서의 `KEYS` 같은 전체 탐색 명령
- 오래 걸리는 Lua Script
- 한 번에 너무 많은 데이터를 반환하는 명령

### 2-8. I/O Multiplexing

Redis는 여러 클라이언트 연결을 효율적으로 다루기 위해 I/O Multiplexing 방식을 사용합니다.

I/O Multiplexing은 하나의 스레드가 여러 소켓의 이벤트를 감시하다가, 읽거나 쓸 준비가 된 소켓을 처리하는 방식입니다.

개념적으로는 다음과 같습니다.

```text
여러 Client Connection
  -> 이벤트 감시
  -> 읽을 준비가 된 요청 처리
  -> 쓸 준비가 된 응답 전송
```

이 방식 덕분에 클라이언트마다 스레드를 하나씩 만들지 않아도 됩니다.  
스레드 생성 비용과 Context Switching 비용을 줄일 수 있습니다.

Redis가 많은 연결을 비교적 효율적으로 처리할 수 있는 이유 중 하나입니다.

### 2-9. Context Switching 비용

CPU는 한 번에 한 스레드만 실행할 수 있는 코어를 여러 개 가지고 있습니다.  
실행 중인 스레드를 바꿀 때는 현재 상태를 저장하고 다음 스레드 상태를 복원해야 합니다.

이 과정을 Context Switching이라고 합니다.

Context Switching이 많아지면 실제 비즈니스 로직을 처리하는 시간보다 스레드를 바꾸는 데 드는 비용이 커질 수 있습니다.

Redis의 이벤트 루프 모델은 이 비용을 줄이는 데 유리합니다.  
요청마다 별도 스레드를 만들지 않고, 이벤트 기반으로 빠르게 처리하기 때문입니다.

물론 Redis도 네트워크 I/O나 백그라운드 작업에서 여러 스레드가 사용될 수 있습니다.  
하지만 핵심 명령 처리 경로가 단순하다는 점이 성능에 유리합니다.

### 2-10. Redis 자료구조와 시간 복잡도

Redis는 단순 문자열 저장소가 아니라 여러 자료구조를 제공합니다.

자료구조에 따라 명령의 시간 복잡도가 달라집니다.

| 자료구조 | 대표 명령 | 일반적인 특징 |
| --- | --- | --- |
| String | `GET`, `SET` | 단순 key-value 접근 |
| Hash | `HGET`, `HSET` | 객체 필드 저장에 유용 |
| List | `LPUSH`, `RPOP` | 큐 형태 처리에 유용 |
| Set | `SADD`, `SISMEMBER` | 중복 없는 집합 |
| Sorted Set | `ZADD`, `ZRANGE` | 점수 기반 정렬 |

Redis가 빠르다고 해서 모든 명령이 항상 같은 속도인 것은 아닙니다.

예를 들어 단순 `GET`은 매우 빠르지만, 큰 Sorted Set에서 넓은 범위를 조회하거나 큰 List 전체를 읽는 작업은 비용이 커질 수 있습니다.

캐싱에서는 보통 `String`이나 `Hash` 형태로 조회 결과를 저장하는 경우가 많습니다.  
이 경우 단순 Key 조회가 중심이므로 Redis의 장점을 잘 활용할 수 있습니다.

정리하면 다음과 같습니다.

> Redis는 빠르지만, 자료구조와 명령의 시간 복잡도를 무시해도 되는 것은 아니다.

### 2-11. Locality: 지역성

캐시가 효과적인 이유 중 하나는 지역성입니다.

지역성은 프로그램이 특정 데이터에 접근하면, 가까운 시점에 같은 데이터나 주변 데이터에 다시 접근할 가능성이 높다는 성질입니다.

대표적으로 두 가지가 있습니다.

#### 시간 지역성

한 번 접근한 데이터는 가까운 미래에 다시 접근될 가능성이 높습니다.

예를 들어 인기 게시글 상세 페이지는 짧은 시간 동안 여러 사용자가 반복해서 조회할 수 있습니다.

#### 공간 지역성

어떤 데이터에 접근하면 그 주변 데이터에도 접근할 가능성이 높습니다.

예를 들어 게시글 목록을 조회한 뒤 각 게시글 상세를 조회하거나, 카테고리 하위 데이터를 함께 조회하는 경우입니다.

Redis 캐싱은 특히 시간 지역성을 활용합니다.

```text
방금 조회한 데이터
  -> Redis에 저장
  -> 가까운 시간 안에 다시 조회되면 빠르게 반환
```

TTL은 이 시간 지역성과 관련이 있습니다.  
얼마나 오래 재사용될 가능성이 있는지를 기준으로 TTL을 잡아야 합니다.

### 2-12. 직렬화와 데이터 크기

Redis에 애플리케이션 객체를 저장하려면 보통 문자열, JSON, 바이너리 형태로 변환해야 합니다.

이 과정을 직렬화라고 합니다.  
Redis에서 다시 읽어 애플리케이션 객체로 바꾸는 과정은 역직렬화입니다.

캐시 값이 너무 크면 다음 비용이 증가합니다.

- 직렬화 CPU 비용
- 역직렬화 CPU 비용
- 네트워크 전송량
- Redis 메모리 사용량
- GC 부담

즉 Redis가 메모리 기반이라 빠르더라도, 큰 데이터를 계속 넣고 빼면 병목이 다른 곳으로 이동할 수 있습니다.

캐시에 저장할 데이터는 가능한 한 작고 명확해야 합니다.

예를 들어 게시글 상세 조회 캐시라면 다음을 고민해야 합니다.

- 화면에 필요한 필드만 저장할 것인가?
- DB Entity 전체를 저장할 것인가?
- DTO 형태로 저장할 것인가?
- 댓글 목록까지 함께 넣을 것인가?
- 사용자별 권한 정보까지 포함할 것인가?

캐시 값이 커질수록 Cache Hit 한 번의 비용도 커집니다.

### 2-13. 성능 병목은 이동한다

Redis를 도입하면 DB 병목이 줄어들 수 있습니다.  
하지만 병목이 완전히 사라지는 것은 아닙니다.

병목은 다른 곳으로 이동할 수 있습니다.

예를 들면 다음과 같습니다.

```text
도입 전 병목: DB CPU, DB I/O, DB Connection
도입 후 병목: Redis Memory, Redis Network, Application CPU, Serialization
```

그래서 Redis 캐싱을 적용한 뒤에도 지표를 봐야 합니다.

- Cache Hit Ratio
- Redis memory usage
- Redis command latency
- Redis connected clients
- Redis network input/output
- Application CPU
- DB query count
- DB connection pool usage

성능 개선은 단순히 "Redis를 붙였다"가 아니라 "어느 병목이 얼마나 줄었는지"로 판단해야 합니다.

## 3. Redis를 캐시로 사용할 때의 기본 구조

Redis는 애플리케이션과 DB 사이에 위치하는 경우가 많습니다.

```text
Client
  -> Application Server
    -> Redis Cache
    -> Database
```

애플리케이션은 요청을 받으면 먼저 Redis에 원하는 데이터가 있는지 확인합니다.

- Redis에 데이터가 있으면 캐시 데이터를 반환함
- Redis에 데이터가 없으면 DB에서 조회한 뒤 Redis에 저장함

여기서 Redis는 원본 저장소가 아니라 보조 저장소입니다.  
따라서 Redis에 데이터가 없더라도 DB를 통해 다시 만들 수 있어야 합니다.

이 관점이 중요합니다.

> 캐시는 없어도 서비스가 동작해야 한다.  
> 단지 캐시가 없으면 느려질 뿐이다.

물론 세션 저장소나 분산 락처럼 Redis 자체가 핵심 상태 저장소 역할을 하는 경우도 있습니다.  
하지만 이번 주차의 캐싱 관점에서는 Redis를 "원본을 빠르게 읽기 위한 임시 저장소"로 보는 것이 핵심입니다.

## 4. 캐싱 전략

### 4-1. Cache Aside

Cache Aside는 실무에서 가장 자주 사용되는 캐싱 전략입니다.  
Lazy Loading이라고도 부릅니다.

흐름은 다음과 같습니다.

1. 애플리케이션이 먼저 Redis를 조회한다.
2. 캐시에 데이터가 있으면 바로 반환한다. Cache Hit
3. 캐시에 데이터가 없으면 DB를 조회한다. Cache Miss
4. DB에서 가져온 데이터를 Redis에 저장한다.
5. 이후 같은 요청은 Redis에서 응답한다.

```text
요청
  -> Redis 조회
    -> 있으면 반환
    -> 없으면 DB 조회
        -> Redis 저장
        -> 반환
```

장점은 구조가 단순하고 필요한 데이터만 캐시에 올라간다는 점입니다.

반면 단점도 있습니다.

- 최초 요청은 여전히 느림
- 캐시에 없는 요청이 몰리면 DB로 트래픽이 집중됨
- DB 데이터 변경 시 캐시 무효화 전략이 필요함

Cache Aside는 다음 데이터에 잘 어울립니다.

- 자주 조회되는 데이터
- 변경보다 조회가 훨씬 많은 데이터
- 잠깐 오래된 값이 보여도 큰 문제가 없는 데이터

예를 들면 게시글 상세, 상품 상세, 인기 게시글 목록, 메인 화면 배너 정보 등이 있습니다.

### 4-2. Write Around

Write Around는 쓰기 작업에서는 캐시를 건드리지 않고 DB에만 저장하는 전략입니다.

흐름은 다음과 같습니다.

```text
쓰기 요청
  -> DB 저장
  -> 캐시는 갱신하지 않음

읽기 요청
  -> Cache Aside 방식으로 필요할 때 캐싱
```

Write Around의 핵심은 "실제로 읽히는 데이터만 캐시에 올리자"입니다.

장점은 불필요한 캐시 저장을 줄일 수 있다는 점입니다.  
한 번 저장되고 다시 조회되지 않는 데이터까지 Redis에 올리지 않아도 됩니다.

단점은 쓰기 직후 읽기에서 오래된 캐시를 볼 수 있다는 점입니다.  
그래서 보통 데이터 수정 시 관련 캐시를 삭제하는 방식과 함께 사용합니다.

```text
데이터 수정
  -> DB 업데이트
  -> 관련 캐시 삭제
```

이렇게 하면 다음 조회에서 Cache Miss가 발생하고, DB의 최신 데이터를 다시 캐시에 저장합니다.

### 4-3. Write Through

Write Through는 쓰기 작업을 할 때 DB와 캐시를 함께 갱신하는 전략입니다.

```text
쓰기 요청
  -> DB 저장
  -> Redis 캐시도 저장
```

장점은 쓰기 직후 캐시에도 최신 데이터가 들어있다는 점입니다.

하지만 단점도 명확합니다.

- 쓰기 작업이 느려질 수 있음
- DB 저장은 성공했는데 Redis 저장은 실패하는 경우를 고려해야 함
- 잘 읽히지 않는 데이터까지 캐시에 저장될 수 있음

그래서 모든 서비스에 무조건 적합한 전략은 아닙니다.  
쓰기보다 읽기가 많고, 쓰기 직후 최신 캐시가 중요할 때 고려할 수 있습니다.

### 4-4. Write Back

Write Back은 먼저 캐시에 쓰고, 나중에 DB에 반영하는 전략입니다.

```text
쓰기 요청
  -> Redis 저장
  -> 일정 시점에 DB 반영
```

쓰기 성능은 좋아질 수 있지만, Redis 장애 시 아직 DB에 반영되지 않은 데이터가 유실될 수 있습니다.

따라서 일반적인 조회 캐시보다는 로그 수집, 버퍼링, 집계성 데이터처럼 약간의 지연 반영을 허용할 수 있는 영역에서 조심스럽게 고려합니다.

## 5. 캐시 히트와 캐시 미스

캐싱을 이해할 때 Cache Hit와 Cache Miss는 기본 용어입니다.

### Cache Hit

요청한 데이터가 캐시에 존재하는 상황입니다.

```text
Redis에 데이터 있음 -> Redis 값 반환
```

Cache Hit가 많을수록 DB 부하가 줄고 응답 속도가 빨라집니다.

### Cache Miss

요청한 데이터가 캐시에 없는 상황입니다.

```text
Redis에 데이터 없음 -> DB 조회 -> Redis 저장 -> 반환
```

Cache Miss가 많으면 Redis를 붙였는데도 DB 부하가 크게 줄지 않습니다.

캐시 효율을 판단할 때는 Hit Ratio를 볼 수 있습니다.

```text
Hit Ratio = Cache Hit 수 / 전체 캐시 조회 수
```

Hit Ratio가 높다는 것은 캐시가 실제로 자주 사용되고 있다는 뜻입니다.  
반대로 Hit Ratio가 낮다면 캐싱 대상이나 TTL 설정이 적절한지 다시 봐야 합니다.

---

## 6. 캐싱에 적합한 데이터

Redis 캐싱은 모든 데이터에 적용하는 것이 아닙니다.

캐싱에 적합한 데이터는 보통 다음 특징을 가집니다.

- 조회 빈도가 높음
- 변경 빈도가 낮음
- 조회 비용이 큼
- 같은 요청이 반복됨
- 약간의 데이터 지연을 허용할 수 있음

예시는 다음과 같습니다.

- 상품 상세 정보
- 게시글 상세 정보
- 인기 게시글 목록
- 카테고리 목록
- 메인 화면 콘텐츠
- 외부 API 호출 결과
- 권한이나 설정처럼 자주 조회되는 메타데이터

반대로 다음 데이터는 신중해야 합니다.

- 잔액, 결제 상태, 재고처럼 정확성이 중요한 데이터
- 사용자별로 거의 재사용되지 않는 데이터
- 변경이 매우 잦은 데이터
- 캐시 키가 너무 많이 생겨 메모리를 빠르게 소모하는 데이터

캐싱할지 말지는 "저장할 수 있느냐"가 아니라 "저장했을 때 비용보다 이득이 크냐"로 판단해야 합니다.

## 7. 캐싱의 대표적인 문제

### 7-1. 데이터 불일치

캐시에서 가장 흔한 문제는 DB와 Redis의 데이터가 달라지는 것입니다.

예를 들어 게시글 제목을 수정했다고 가정합니다.

```text
DB: 수정된 제목
Redis: 수정 전 제목
```

이 상태에서 조회 API가 Redis 값을 반환하면 사용자는 수정 전 제목을 보게 됩니다.

이 문제를 줄이는 방법은 크게 세 가지입니다.

- TTL을 짧게 둔다.
- 데이터 변경 시 관련 캐시를 삭제한다.
- 데이터 변경 시 캐시도 함께 갱신한다.

어떤 방식이 정답인지는 서비스 요구사항에 따라 달라집니다.

### 7-2. 메모리 한계

Redis는 메모리 기반 저장소입니다.  
따라서 캐시를 많이 저장할수록 메모리 비용이 증가합니다.

캐시 대상이 너무 많거나 TTL이 너무 길면 Redis 메모리가 빠르게 찰 수 있습니다.

메모리 한계를 고려할 때는 다음을 봐야 합니다.

- 캐시 키 개수
- 값의 평균 크기
- TTL 길이
- 만료 정책
- eviction 정책

Redis를 캐시로 쓴다면 "언젠가 지워질 데이터"라는 전제를 두고 설계해야 합니다.

### 7-3. Cache Stampede

Cache Stampede는 특정 캐시가 만료된 순간 많은 요청이 동시에 DB로 몰리는 문제입니다.

예를 들어 인기 상품 정보 캐시가 정확히 10시에 만료되었다고 가정합니다.  
10시에 많은 사용자가 동시에 해당 상품을 조회하면 모든 요청이 Cache Miss가 되고 DB로 향할 수 있습니다.

해결 방법은 다음과 같습니다.

- 하나의 요청만 DB를 조회하도록 락을 사용한다.
- TTL에 랜덤 값을 섞어 만료 시점을 분산한다.
- 만료되기 전에 백그라운드에서 미리 갱신한다.
- 아주 인기 있는 데이터는 별도 워밍업을 한다.

### 7-4. Cache Penetration

Cache Penetration은 존재하지 않는 데이터를 계속 조회해 캐시를 우회하고 DB까지 요청이 도달하는 문제입니다.

예를 들어 존재하지 않는 상품 ID를 반복해서 조회하면 Redis에는 데이터가 없고, 매번 DB 조회가 발생합니다.

해결 방법은 다음과 같습니다.

- 존재하지 않는 결과도 짧은 TTL로 캐싱한다. null caching
- 요청 파라미터를 검증한다.
- Bloom Filter 같은 자료구조로 존재 가능성을 먼저 검사한다.

### 7-5. Cache Snowball

Cache Snowball은 많은 캐시가 비슷한 시점에 한꺼번에 만료되면서 DB 부하가 급증하는 문제입니다.

Cache Stampede가 특정 인기 키 하나의 문제라면, Cache Snowball은 여러 키가 동시에 만료되는 문제에 가깝습니다.

해결 방법은 다음과 같습니다.

- TTL을 동일하게 주지 않고 분산한다.
- 캐시 생성 시 랜덤 TTL을 추가한다.
- 중요한 캐시는 미리 재생성한다.
- Redis 장애 상황에서 DB를 보호할 fallback 정책을 둔다.

## 8. TTL의 역할

TTL은 Time To Live의 약자로, 캐시 데이터가 살아 있는 시간을 의미합니다.

Redis에서는 키마다 TTL을 설정할 수 있고, 시간이 지나면 해당 키는 만료됩니다.

캐싱에서 TTL이 중요한 이유는 두 가지입니다.

1. 오래된 데이터가 무한히 남는 것을 막는다.
2. Redis 메모리가 계속 증가하는 것을 막는다.

TTL이 너무 짧으면 Cache Miss가 자주 발생해 캐시 효과가 줄어듭니다.  
TTL이 너무 길면 오래된 데이터가 사용자에게 노출될 가능성이 커집니다.

따라서 TTL은 성능과 정합성 사이의 타협점입니다.

예를 들어 다음처럼 생각할 수 있습니다.

| 데이터 | 변경 빈도 | 정합성 중요도 | TTL 방향 |
| --- | --- | --- | --- |
| 게시글 상세 | 중간 | 중간 | 수분 단위 |
| 인기 게시글 목록 | 높음 | 낮음 | 짧게 |
| 카테고리 목록 | 낮음 | 낮음 | 길게 |
| 결제 상태 | 높음 | 매우 높음 | 캐싱 신중 |
| 외부 API 결과 | 낮음 | 중간 | API 정책에 맞춤 |

정답 TTL은 없습니다.  
서비스 특성, 트래픽, 데이터 변경 빈도, 사용자가 허용할 수 있는 지연 시간을 함께 고려해야 합니다.

## 9. 캐시 무효화

캐시 무효화는 원본 데이터가 변경되었을 때 기존 캐시를 더 이상 사용하지 않도록 만드는 작업입니다.

캐시 무효화 방식은 크게 두 가지입니다.

### 삭제 방식

DB를 수정한 뒤 관련 캐시를 삭제합니다.

```text
DB 수정
  -> Redis 캐시 삭제
  -> 다음 조회에서 DB 조회 후 캐시 재생성
```

장점은 단순하다는 것입니다.  
캐시를 직접 최신 값으로 맞추지 않고 삭제만 하면 됩니다.

단점은 다음 조회에서 Cache Miss가 발생한다는 점입니다.

### 갱신 방식

DB를 수정한 뒤 캐시도 최신 값으로 갱신합니다.

```text
DB 수정
  -> Redis 캐시 갱신
```

장점은 다음 조회가 바로 Cache Hit 될 수 있다는 것입니다.

단점은 DB 업데이트와 Redis 업데이트 중 하나만 성공하는 상황을 고려해야 한다는 점입니다.  
또한 캐시 값 생성 로직이 복잡하면 갱신 방식이 오히려 부담이 될 수 있습니다.

실무에서는 단순성과 안정성 때문에 삭제 방식이 많이 사용됩니다.  
특히 Cache Aside와 함께 사용할 때 자연스럽습니다.

## 10. Spring Boot와 Redis 캐싱을 이론적으로 이해하기

Spring Boot에서 Redis 캐싱을 사용할 때 핵심은 "비즈니스 로직 앞뒤에 캐시 계층을 붙이는 것"입니다.

Spring에서는 보통 다음 구조로 이해할 수 있습니다.

```text
Controller
  -> Service
    -> Cache
    -> Repository
    -> Database
```

중요한 관심사는 코드 작성 방법이 아니라 역할 분리입니다.

- Controller는 요청과 응답을 담당한다.
- Service는 비즈니스 흐름을 담당한다.
- Repository는 DB 접근을 담당한다.
- Cache는 반복 조회 결과를 빠르게 반환하는 보조 계층이다.

Spring Cache 추상화를 사용하면 애플리케이션 코드는 "캐시 저장소가 Redis인지 로컬 메모리인지"를 어느 정도 분리해서 생각할 수 있습니다.

이론적으로 보면 Spring의 캐싱은 다음 질문을 선언적으로 표현하는 방식입니다.

- 어떤 메서드의 결과를 캐싱할 것인가?
- 캐시 키는 무엇으로 만들 것인가?
- 언제 캐시를 삭제할 것인가?
- 어떤 조건에서는 캐싱하지 않을 것인가?

Spring Boot에서 Redis를 캐시로 쓸 때 특히 중요한 개념은 캐시 키 설계입니다.

예를 들어 상품 상세 조회라면 키는 다음처럼 도메인과 식별자가 드러나야 합니다.

```text
product:detail:1
post:detail:100
user:profile:7
```

키를 잘 설계해야 운영 중에 어떤 데이터가 캐시되어 있는지 추적하기 쉽고, 특정 도메인의 캐시를 삭제하거나 점검하기도 쉬워집니다.

---

## 11. NestJS와 Redis 캐싱을 이론적으로 이해하기

NestJS에서도 핵심 구조는 Spring Boot와 크게 다르지 않습니다.

```text
Controller
  -> Service
    -> Cache
    -> Repository 또는 외부 API
```

NestJS는 모듈 구조를 기반으로 애플리케이션을 구성합니다.  
Redis 캐시는 특정 도메인 내부에 둘 수도 있고, 여러 도메인에서 공통으로 사용하는 캐시 계층으로 분리할 수도 있습니다.

이론적으로 중요한 점은 Redis가 특정 프레임워크에 종속된 기술이 아니라는 것입니다.

Spring Boot든 NestJS든 캐싱의 본질은 같습니다.

1. 요청이 들어온다.
2. 캐시 키를 만든다.
3. Redis에서 먼저 조회한다.
4. 있으면 반환한다.
5. 없으면 원본 저장소를 조회한다.
6. 결과를 Redis에 저장한다.
7. 응답한다.

프레임워크가 달라져도 바뀌지 않는 것은 캐싱 전략입니다.

- Cache Aside를 쓸 것인가?
- TTL은 얼마로 둘 것인가?
- 캐시 키는 어떻게 만들 것인가?
- 데이터 변경 시 캐시는 삭제할 것인가, 갱신할 것인가?
- Redis 장애 시 DB로 fallback할 것인가?

따라서 이번 주차에서 Spring Boot와 NestJS를 함께 다루더라도, 발표에서 강조해야 할 부분은 프레임워크별 문법보다 캐싱 구조와 의사결정 기준입니다.

---

## 12. 캐싱 전에 먼저 확인할 것

캐싱은 강력하지만 항상 첫 번째 해결책은 아닙니다.

API가 느릴 때 바로 Redis를 붙이기 전에 먼저 확인해야 할 것이 있습니다.

### 12-1. SQL 튜닝

느린 API의 원인이 비효율적인 SQL이라면 캐싱보다 SQL 튜닝이 먼저입니다.

예를 들면 다음 문제가 있을 수 있습니다.

- 인덱스가 없음
- 불필요한 전체 테이블 스캔
- N+1 문제
- 너무 많은 컬럼 조회
- 불필요한 조인
- 페이징 방식 문제

이런 문제를 해결하지 않고 캐시만 붙이면 Cache Hit 상황에서는 빨라질 수 있지만, Cache Miss 상황에서는 여전히 느립니다.

### 12-2. 캐싱 대상이 맞는지 확인

모든 API가 캐싱에 적합하지는 않습니다.

다음 질문에 답할 수 있어야 합니다.

- 같은 요청이 반복되는가?
- 조회가 쓰기보다 훨씬 많은가?
- 데이터가 조금 늦게 반영되어도 괜찮은가?
- 캐시 키 개수가 과도하게 늘어나지 않는가?
- Redis 장애 시 원본 저장소로 처리 가능한가?

이 질문에 답하기 어렵다면 캐시 도입 전에 요구사항을 더 명확히 해야 합니다.

### 12-3. 측정

캐싱은 감으로 적용하면 안 됩니다.

도입 전후로 다음 지표를 비교해야 합니다.

- API 응답 시간
- DB Query 수
- DB CPU 사용률
- DB 커넥션 사용량
- Redis 메모리 사용량
- Cache Hit Ratio

성능 개선은 "빨라진 것 같다"가 아니라 수치로 확인해야 합니다.

## 13. Redis 캐싱 운영 시 주의점

### 13-1. Redis 장애를 가정하기

캐시로 사용하는 Redis는 장애가 나도 서비스가 완전히 멈추지 않도록 설계하는 것이 좋습니다.

일반적인 조회 캐시라면 Redis 장애 시 DB로 fallback할 수 있어야 합니다.

```text
Redis 조회 실패
  -> 로그 기록
  -> DB 조회
  -> 응답
```

다만 Redis 장애 상황에서 모든 트래픽이 DB로 몰릴 수 있으므로, DB 보호 정책도 같이 필요합니다.

예를 들면 다음과 같습니다.

- timeout을 짧게 설정
- fallback 시 rate limit 적용
- 인기 데이터는 사전 캐싱
- 장애 상황에서 일부 기능 제한

### 13-2. 키 네이밍 규칙

캐시 키는 운영 중 확인하기 쉬워야 합니다.

좋은 키는 보통 다음 정보를 포함합니다.

- 도메인
- 용도
- 식별자
- 필요하다면 버전

예시는 다음과 같습니다.

```text
post:detail:100
post:list:popular:v1
user:profile:7
category:tree:v1
```

키 네이밍이 정리되어 있으면 장애 분석, 캐시 삭제, 모니터링이 쉬워집니다.

### 13-3. 직렬화 비용

Redis는 네트워크를 통해 데이터를 주고받습니다.  
애플리케이션 객체를 Redis에 저장하려면 보통 문자열, JSON, 바이너리 형태로 직렬화해야 합니다.

데이터가 너무 크면 다음 비용이 커집니다.

- 직렬화 비용
- 역직렬화 비용
- 네트워크 전송 비용
- Redis 메모리 사용량

따라서 캐시는 "필요한 데이터만 작게 저장하는 것"이 좋습니다.

### 13-4. 큰 키와 많은 키

Redis에서는 하나의 큰 값도 문제고, 너무 많은 키도 문제입니다.

하나의 키에 너무 큰 데이터를 넣으면 읽고 쓰는 동안 Redis와 네트워크에 부담이 됩니다.  
반대로 너무 많은 키를 만들면 메모리 관리와 만료 관리가 어려워질 수 있습니다.

캐시 설계 시에는 다음을 고려해야 합니다.

- 값의 최대 크기
- 키 개수 증가 속도
- TTL 적용 여부
- 사용자별 캐시가 과도하게 늘어나는지

## 14. 발표에서 강조할 비교

### Redis vs RDB

| 항목 | Redis | RDB |
| --- | --- | --- |
| 주요 목적 | 빠른 데이터 접근 | 안정적인 원본 데이터 관리 |
| 저장 구조 | Key-Value 중심 | Table, Row, Relation |
| 저장 위치 | 주로 RAM | Disk + Buffer Pool |
| 강점 | 빠른 조회, TTL, 간단한 자료구조 | 트랜잭션, 정합성, 복잡한 질의 |
| 약점 | 메모리 비용, 정합성 관리 필요 | 상대적으로 무거운 조회 비용 |
| 대표 용도 | 캐시, 세션, 랭킹, 카운터 | 원본 데이터 저장 |

Redis와 RDB는 경쟁 관계라기보다 역할이 다릅니다.

RDB는 원본 데이터의 정합성을 책임지고, Redis는 반복 조회를 빠르게 처리하는 보조 계층으로 쓰는 경우가 많습니다.

### 성능과 일관성의 Trade-off

캐싱은 성능을 얻는 대신 일관성 관리 비용을 추가합니다.

```text
성능을 높이고 싶다
  -> 캐시를 오래 유지한다
  -> Cache Hit가 늘어난다
  -> 오래된 데이터가 보일 가능성이 커진다

정합성을 높이고 싶다
  -> 캐시를 자주 삭제하거나 TTL을 짧게 둔다
  -> 최신 데이터에 가까워진다
  -> Cache Miss가 늘어난다
```

결국 캐시 설계는 이 균형점을 찾는 일입니다.

## 15. 최종 정리

이번 주차의 핵심은 다음과 같습니다.

1. Redis는 RAM 기반의 빠른 Key-Value 저장소다.
2. Redis 캐싱은 DB 조회 비용을 줄이고 응답 속도를 높이는 데 효과적이다.
3. 가장 대표적인 전략은 Cache Aside이다.
4. 캐시는 원본 저장소가 아니라 원본 앞의 빠른 보조 계층으로 이해해야 한다.
5. 캐싱은 성능과 데이터 일관성 사이의 Trade-off를 만든다.
6. TTL, 캐시 무효화, 키 네이밍, 장애 fallback을 함께 설계해야 한다.
7. 캐싱 전에 SQL 튜닝과 병목 측정이 먼저 필요하다.
8. Spring Boot와 NestJS에서도 캐싱의 본질은 동일하다. 프레임워크 문법보다 캐싱 전략과 설계 기준이 더 중요하다.
9. Redis가 빠른 이유는 RAM뿐 아니라 RDB의 복잡한 조회 경로를 단순한 Key-Value 조회로 우회하기 때문이다.
