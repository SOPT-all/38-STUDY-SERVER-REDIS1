# Week 01 - 유서연

## Redis란?

Redis는 **Remote Dictionary Server**의 약자로,
- Remote: 원격에 위치하고
- Dictionary: 데이터를 Key-Value 형태로 저장하는 NoSQL 데이터 저장소이면서,
- Server: 데이터를 메모리(RAM)에 저장하는 In-Memory 데이터 저장소이다.

## Redis를 사용하는 이유

Redis를 사용하는 가장 큰 이유는 **빠른 조회 속도**와 **DB 부하 감소**이다.
서비스에서 같은 데이터가 짧은 시간 동안 여러 번 조회될 때, 매번 데이터베이스에서 조회하면 DB에 부하가 쌓이고 응답 속도도 느려진다.
하지만, 이때 Redis를 활용하면 자주 사용하는 데이터를 메모리에 저장해두고 빠르게 꺼내 쓸 수 있다.

- **Redis 사용 전**: Client → Server → DB
- **Redis 사용 후**: Client → Server → Redis -> (Redis에 존재하지 않으면) DB

즉, Redis는 자주 조회되거나 일정 시간 동안만 필요한 데이터를 빠르게 처리하기 위한 보조 저장소이다.

## Redis의 장점

1. **빠른 조회 속도**
   
   위에서 언급했던 것처럼, 데이터를 메모리에 저장하기 때문에 읽기와 쓰기 속도가 빠르다.

2. **사용이 간편한 Key-Value 구조**  
   
   Key를 통해 Value를 바로 조회할 수 있기 때문에 구조가 단순하고 사용하기 쉽다.

   ```bash
   seoyeon:name → "seoyeon yu"
   ```
  

3. **TTL 설정 가능**  
   
   TTL은 **Time To Live**의 약자로, 데이터가 살아있는 시간을 의미한다.
   Redis는 이 TTL을 데이터마다 지정할 수 있는데, 예를 들어
   ```bash
   set auth:email:code 123456 ex 180
   ```
   위 코드에 따르면 해당 이메일 인증번호 데이터는 180초가 지난 뒤 자동으로 삭제된다.
   Redis는 메모리 기반 저장소이기 때문에 메모리 공간이 한정되어 있다.  
   따라서 모든 데이터를 Redis에 저장하기보다는,
   TTL을 활용해 자주 사용하는 데이터만 일정 시간 동안 저장해두는 방식으로 사용하는 것이 좋다.

4. **다양한 자료구조 지원**  
   
   단순 문자열뿐만 아니라 List, Set, Hash, Sorted Set 등 다양한 자료구조를 사용할 수 있다.

   | 자료구조 | 설명 | 사용 예시 |
   | --- | --- | --- |
   | String | 가장 기본적인 Key-Value 저장 | 인증번호, 토큰, 단순 캐시 |
   | List | 순서가 있는 값 목록 | 최근 본 상품, 작업 큐 |
   | Set | 중복 없는 값 목록 | 좋아요한 사용자 목록 |
   | Sorted Set | 점수를 기준으로 정렬되는 Set | 랭킹, 인기 게시글 |
   | Hash | 하나의 Key 안에 여러 field-value 저장 | 사용자 프로필, 객체 형태 데이터 |

5. **영속성 지원**
   
   Redis는 휘발성 메모리에 저장되다보니 서비가 꺼지면 데이터가 사라질 수 있다.</br>
   하지만, Redis는 데이터를 디스크에도 따로 저장하는 방식을 차용하여 서버가 꺼져도 데이터가 유지되는 영속성을 가진다.

   - **휘발성**: 데이터를 저장하고 있던 프로그램이나 서버가 종료되면 데이터가 사라지는 특성
   - **영속성**: 데이터를 생성한 프로그램이나 서버가 종료되더라도 데이터가 사라지지 않고, 파일 시스템이나 관계형 데이터베이스(RDB) 같은 영구 저장소에 저장되어 유지되는 특성

   Redis가 데이터를 저장하는 방식은 다음과 같다.

   | 방식 | 설명 |
   | --- | --- |
   | RDB | 특정 시점의 데이터를 스냅샷으로 저장 |
   | AOF | 모든 쓰기 작업을 로그로 기록 |
   | RDB + AOF | 스냅샷과 로그 방식을 함께 사용 |

6. **원자적 연산 지원**  
   
   원자적 연산이란 하나의 명령이 실행되는 도중에 다른 명령이 끼어들지 않는다는 뜻이다.
   예를 들어, 조회수 증가처럼 여러 사용자가 동시에 같은 값을 수정하는 상황을 가정했을 때, 여러 요청이 들어오면 값이 꼬일 가능성이 있다.
   하지만, Redis는 명령을 하나씩 처리하기 때문에 중간에 값이 꼬일 가능성이 낮다.

7. **복제와 클러스터링 지원**  
   
   Redis는 서버를 여러 대로 구성할 수 있기 때문에, 장애에 대비하거나, 더 많은 요청과 데이터를 처리할 수 있다.

   - Replication

   Redis 서버를 복제하는 방식
   일반적으로 주 서버는 쓰기 요청을 처리하고, 복제 서버는 읽기 요청을 분산해서 처리하는 데 활용할 수 있다.
   이를 통해 읽기 요청을 여러 서버로 나눌 수 있고, 주 서버에 문제가 생겼을 때 복제 서버를 활용할 수 있다.

   - Redis Cluster

   데이터를 여러 Redis 서버에 나누어 분산 저장하는 방식
   데이터를 분산 저장하기 때문에 더 많은 데이터를 처리할 수 있고, 쓰기 요청도 여러 노드로 나눌 수 있다.

   ```text
   Redis Cluster

   Node 1 → 일부 데이터 저장
   Node 2 → 일부 데이터 저장
   Node 3 → 일부 데이터 저장
   ```

## Redis의 주요 사용 사례

| 사용 사례 | 설명 | 예시 |
| --- | --- | --- |
| 캐싱<br>(Caching) | 자주 조회되는 데이터를 Redis에 저장해 DB 대신 빠르게 응답한다. | 인기 게시글, 상품 정보 |
| 세션 관리<br>(Session Management) | 로그인 세션을 Redis에 저장해 여러 서버가 함께 사용할 수 있게 한다. | 로그인 세션, refresh token |
| 실시간 분석 및 통계<br>(Real-time Analytics) | 빠르게 변하는 수치를 저장하고 조회할 수 있다. | 조회수, 클릭 수 |
| 메시지 큐<br>(Message Queue) | 처리할 작업을 Redis에 쌓아두고 순서대로 꺼내 처리한다. | 이메일 발송, 알림 전송 |
| 지리공간 인덱싱<br>(Geospatial Indexing) | 위치 데이터를 저장하고 특정 위치 주변 데이터를 조회한다. | 주변 매장 찾기 |
| 속도 제한<br>(Rate Limiting) | 일정 시간 동안의 요청 횟수를 저장해 과도한 요청을 제한한다. | 인증번호 요청 제한, API 호출 제한 |
| 실시간 채팅 및 메시징<br>(Real-time Chat and Messaging) | Pub/Sub을 활용해 메시지를 실시간으로 전달한다. | 채팅, 실시간 알림 |

## Redis 기본 명령어

- **데이터 저장하기 `set`**

```bash
# set [key 이름] [value]
$ set seoyeon:name "seoyeon yu"
```

- **데이터 조회하기 `get`**

Key로 Value 값을 조회할 수 있다.
존재하지 않는 Key를 조회하면 `(nil)`이 출력된다.

```bash
# get [key 이름]
$ get seoyeon:name
```

- **데이터 삭제하기 `del`**

```bash
# del [key 이름]
$ del seoyeon:name
```

- **만료시간 설정하기 `ex`**

```bash
# set [key 이름] [value] ex [만료 시간(초)]
$ set seoyeon:pet dog ex 30
```

`ex` 옵션을 사용하면 데이터가 일정 시간이 지난 뒤 자동으로 삭제되도록 설정할 수 있다.

- **만료시간 확인하기 `ttl`**

```bash
# ttl [key 이름]
$ ttl seoyeon:pet
```

`ttl`은 Key의 만료 시간이 몇 초 남았는지 확인할 때 사용하는 명령어이다.

| 반환값 | 의미 |
| --- | --- |
| 양수 | 만료까지 남은 시간 |
| -1 | Key는 존재하지만 만료 시간이 설정되어 있지 않음 |
| -2 | Key가 존재하지 않음 |

- **모든 데이터 삭제하기 `flushall`**

```bash
$ flushall
```

## Key 네이밍 컨벤션

Redis Key는 콜론(`:`)을 활용해 계층적으로 의미를 구분해서 작성하는 경우가 많다.

```bash
도메인:식별자:데이터종류
```

```bash
users:100:profile
auth:email:code
```

- `users:100:profile`: 100번 사용자의 프로필 정보
- `auth:email:code`: 이메일 인증번호 정보

---

## Key 네이밍 컨벤션의 장점

1. **가독성**  

   Key 이름만 봐도 어떤 데이터인지 의미와 용도를 쉽게 파악할 수 있다.

2. **일관성**  

   정해둔 규칙대로 Key를 작성하면 코드 스타일이 통일되고, 유지보수가 쉬워진다.

3. **검색 및 필터링 용이성**  

   패턴을 활용해 특정 유형의 Key를 쉽게 찾을 수 있다.

4. **확장성**  

   기능이나 데이터가 늘어나더라도 Key 이름이 겹칠 가능성이 줄어들어 안정적으로 관리할 수 있다.
