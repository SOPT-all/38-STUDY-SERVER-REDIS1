# Week 01 - 김규일

## Redis란?

Redis는 `Remote Dictionary Server`의 약자로 메모리 기반의 오픈소스 키-값 저장소입니다.  
단순한 문자열 저장만 하는 것이 아니라 `String`, `List`, `Set`, `Sorted Set`, `Hash`, `Stream` 같은 다양한 자료구조를 지원합니다.

보통 Redis는 다음과 같은 특징 때문에 많이 사용됩니다.

- 메모리 기반이라 읽기/쓰기 속도가 매우 빠름
- 데이터 구조가 단순해서 캐시, 세션 저장소, 랭킹 시스템 등에 잘 맞음
- TTL(Time To Live) 기능을 제공해서 만료되는 데이터를 다루기 쉬움
- Pub/Sub, Lua Script, Transaction 등 부가 기능도 제공함

Redis는 "빠르게 읽고 쓰고 짧은 시간 유지되는 데이터를 다루는 데 강한 저장소"라고 정리할 수 있습니다.

## Redis의 장점

- 속도가 빠름  
  디스크가 아니라 메모리를 주로 사용하기 때문에 응답 속도가 매우 빠릅니다.
- 다양한 자료구조 지원  
  단순 캐시를 넘어서 큐, 랭킹, 카운터, 세션 저장 등 다양한 패턴을 쉽게 구현할 수 있습니다.
- TTL 지원이 편리함  
  캐시 만료, 인증 코드 만료, 세션 만료 같은 기능을 구현하기 좋습니다.
- 운영이 비교적 단순함  
  기본 사용법이 직관적이고 CLI로 바로 확인하기 쉬워 학습 난이도가 상대적으로 낮습니다.
- 높은 활용도  
  웹 서비스, 게임, 실시간 시스템, 분산 락 등 실제 서비스에서 쓰임새가 많습니다.

## Redis의 단점

- 메모리 비용이 큼  
  디스크 기반 DB보다 메모리가 비싸기 때문에 대용량 데이터를 장기간 저장하는 용도에는 부담이 큽니다.
- 영속성 관점에서 주의가 필요함  
  Redis는 메모리 기반이므로 설정에 따라 장애 시 일부 데이터 유실 가능성을 고려해야 합니다.
- 복잡한 관계형 질의에 약함  
  조인, 복잡한 조건 검색 같은 기능은 관계형 데이터베이스보다 불리합니다.
- 데이터 구조 설계를 잘해야 함  
  단순히 "빠르다"만 보고 쓰면 키 설계, 만료 정책, 메모리 사용량 문제로 운영이 어려워질 수 있습니다.

## Redis 사용 사례

### 1. 캐시(Cache)

가장 대표적인 사용 사례입니다.  
DB에서 자주 조회되는 데이터를 Redis에 캐싱해 두면 응답 시간을 줄이고 DB 부하를 낮출 수 있습니다.

예시

- 상품 상세 정보 캐시
- 인기 게시글 목록 캐시
- 사용자 프로필 조회 결과 캐시

### 2. 세션 저장소(Session Store)

로그인 상태나 사용자 인증 정보를 일정 시간 동안 저장하는 데 자주 사용됩니다.

예시

- 로그인 세션 저장
- 인증 토큰/리프레시 토큰 관리
- OTP 인증 코드 저장

### 3. 카운터(Counter)

증가/감소 연산이 빠르기 때문에 조회 수, 좋아요 수, 방문자 수 같은 기능 구현에 적합합니다.

예시

- 게시글 조회 수 증가
- 실시간 접속자 수 집계
- API 호출 횟수 제한(rate limiting)

### 4. 큐(Queue)와 작업 처리

List 또는 Stream 자료구조를 이용해 간단한 작업 큐를 만들 수 있습니다.

예시

- 비동기 알림 발송 대기열
- 로그/이벤트 수집
- 백그라운드 작업 분배

### 5. 랭킹 시스템

`Sorted Set`을 사용하면 점수 기반 정렬이 쉬워서 리더보드 구현에 자주 쓰입니다.

예시

- 게임 점수 랭킹
- 인기 검색어 순위
- 실시간 인기 게시글 정렬

### 6. 분산 락(Distributed Lock)

여러 서버가 같은 자원에 동시에 접근하지 못하게 제어할 때 사용합니다.

예시

- 중복 결제 방지
- 스케줄러 중복 실행 방지
- 재고 차감 동시성 제어

## Redis CLI에서 자주 쓰는 명령어

### 접속 및 기본 확인

```bash
redis-cli
PING
INFO
DBSIZE
```

- `PING`: Redis 서버가 정상 응답하는지 확인
- `INFO`: 서버 상태, 메모리, 통계 등 확인
- `DBSIZE`: 현재 DB에 저장된 키 개수 확인

### 문자열(String)

```bash
SET user:1:name "kim"
GET user:1:name
SET login:token "abc123" EX 60
INCR page:view:1
DECR stock:item:1
```

- `SET`: 값 저장
- `GET`: 값 조회
- `EX`: 초 단위 TTL 설정
- `INCR` / `DECR`: 숫자 증가/감소

### 키 관리

```bash
KEYS *
SCAN 0
EXISTS user:1:name
DEL user:1:name
EXPIRE login:token 30
TTL login:token
```

- `KEYS *`: 전체 키 확인, 운영 환경에서는 부담이 커서 주의 필요
- `SCAN 0`: 키를 점진적으로 조회할 때 사용
- `EXISTS`: 키 존재 여부 확인
- `DEL`: 키 삭제
- `EXPIRE`: 기존 키에 만료 시간 설정
- `TTL`: 남은 만료 시간 확인

### 해시(Hash)

```bash
HSET user:1 name "kim" age 30
HGET user:1 name
HGETALL user:1
```

- `HSET`: 해시 필드 저장
- `HGET`: 특정 필드 조회
- `HGETALL`: 전체 필드 조회

### 리스트(List)

```bash
LPUSH queue:email "job1"
RPUSH queue:email "job2"
LPOP queue:email
LRANGE queue:email 0 -1
```

- `LPUSH` / `RPUSH`: 리스트 앞/뒤에 데이터 삽입
- `LPOP`: 앞에서 데이터 꺼내기
- `LRANGE`: 범위 조회

### 셋(Set) / 정렬된 셋(Sorted Set)

```bash
SADD tags "redis" "cache" "db"
SMEMBERS tags

ZADD ranking 100 kim
ZADD ranking 80 lee
ZRANGE ranking 0 -1 WITHSCORES
ZREVRANGE ranking 0 2 WITHSCORES
```

- `SADD`: Set에 값 추가
- `SMEMBERS`: Set 전체 조회
- `ZADD`: 점수와 함께 Sorted Set에 저장
- `ZRANGE`: 오름차순 조회
- `ZREVRANGE`: 내림차순 조회

## Redis CLI를 AI와 같이 쓰면 편한 점

Redis CLI는 이미 AI가 비교적 잘 알고 있는 CLI 중 하나이기 때문에 프로젝트에서 Redis 관련 작업을 할 때 AI에게 CLI 기반으로 작업을 시키면 효율이 올라갈 수 있습니다.

예를 들면 다음과 같은 작업을 빠르게 맡길 수 있습니다.

- 특정 키가 실제로 어떻게 저장되어 있는지 확인
- TTL이 제대로 설정되었는지 점검
- 캐시 키 패턴을 조회해서 문제 원인 파악
- 랭킹 데이터나 세션 데이터를 CLI로 직접 검증
- 테스트용 Redis 데이터 생성 및 삭제

특히 Redis는 `GET`, `HGETALL`, `TTL`, `SCAN`, `LRANGE`, `ZRANGE`처럼 상태를 바로 확인할 수 있는 명령어가 많아서 AI가 필요한 명령어를 조합해 빠르게 점검 절차를 제안하거나 직접 CLI 작업을 수행하기 좋습니다.

## TTL이란?

TTL은 `Time To Live`의 약자로 "데이터가 살아 있을 수 있는 남은 시간"을 뜻합니다.  
Redis에서는 특정 키에 만료 시간을 설정할 수 있고 시간이 지나면 해당 키는 자동으로 삭제됩니다.

예를 들어 아래 명령은 `login:token` 키를 60초 뒤에 만료시킵니다.

```bash
SET login:token "abc123" EX 60
TTL login:token
```

TTL이 필요한 대표 사례는 다음과 같습니다.

- 로그인 세션 만료
- 인증 코드 3분 제한
- 캐시 데이터 10분 보관
- API 요청 횟수 제한 시간 관리

## TTL의 작동 원리

Redis의 TTL은 단순히 "시간이 되면 무조건 즉시 삭제"되는 방식으로만 이해하면 부족합니다.  
실제로는 Redis가 만료 시간을 별도로 관리하면서 키를 접근하거나 주기적으로 확인할 때 만료된 키를 정리합니다.

핵심 흐름은 다음과 같습니다.

### 1. 키와 만료 시간을 함께 저장

Redis는 값 자체와 별개로 "이 키는 언제 만료되는지"에 대한 정보를 저장합니다.

예시

- `user:session:1 -> 값`
- `user:session:1 -> 만료 시각`

즉 TTL은 키에 대한 부가 메타데이터처럼 관리됩니다.

### 2. Lazy Expiration

어떤 키를 조회했을 때 이미 만료 시간이 지났다면 그 순간 Redis가 해당 키를 만료된 것으로 판단하고 삭제합니다.
사용자가 접근하는 시점에 정리되는 방식입니다.

장점

- 모든 키를 계속 검사하지 않아도 되어 효율적임

한계

- 아무도 접근하지 않는 만료 키는 바로 안 지워질 수도 있음

### 3. Active Expiration

Redis는 주기적으로 일부 만료 대상 키를 샘플링해서 이미 만료된 키를 정리합니다.
접근이 없어도 백그라운드에서 일정량의 expired key를 청소합니다.

장점

- Lazy Expiration만 사용할 때 생길 수 있는 "안 쓰는 만료 키가 계속 남아 있는 문제"를 줄여 줌

### 4. 완전히 실시간 보장은 아님

TTL이 0초가 되는 순간 운영체제 알람처럼 정확히 즉시 삭제된다고 생각하면 안 됩니다.  
실제 삭제 시점은 접근 시점 또는 Redis의 active expiration 주기에 따라 조금 달라질 수 있습니다.

그래도 애플리케이션 입장에서는 "지정 시간이 지나면 더 이상 유효하지 않다"는 의미로 사용하면 충분합니다.

## TTL 관련해서 같이 알아두면 좋은 점

- `TTL key` 결과가 `-1`이면 만료 시간이 설정되지 않은 키입니다.
- `TTL key` 결과가 `-2`이면 키가 존재하지 않거나 이미 만료된 상태입니다.
- `EXPIRE key 60`은 기존 키에 TTL을 설정합니다.
- `SET key value EX 60`은 값을 저장하면서 동시에 TTL을 설정합니다.
- 만료 시간이 필요한 데이터는 키 네이밍도 같이 설계하는 것이 좋습니다.  
  예: `session:user:1`, `otp:signup:01012345678`, `rate-limit:user:1`

## 정리

Redis는 메모리 기반의 매우 빠른 키-값 저장소이며 캐시/세션/카운터/랭킹/큐 같은 기능 구현에 특히 강합니다.  
그리고 TTL 기능 덕분에 "일정 시간이 지나면 자동으로 사라져야 하는 데이터"를 다루기 좋습니다.

이번 주차에서 Redis를 학습하면서 느낀 핵심은 아래 두 가지입니다.

- Redis는 단순 캐시가 아니라 다양한 자료구조를 가진 실용적인 저장소다.
- TTL을 이해하면 Redis가 왜 세션, 인증, 캐시 관리에 강한지 자연스럽게 이해할 수 있다.
